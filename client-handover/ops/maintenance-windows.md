# חלונות תחזוקה (Maintenance Windows)

## 1. עקרונות

- תחזוקה מתבצעת **בשעות הפעילות הנמוכה ביותר** של הלקוחות
- **הודעה מראש** של 7 ימים לפחות לתחזוקה רגילה, 48 שעות לתחזוקת חירום
- **שקיפות מלאה** — Status Page מתעדכן בזמן אמת
- **Rollback Plan** מוכן לכל פעולה

## 2. סוגי תחזוקה

### 2.1 תחזוקה תכופה (Frequent)
- **תדירות**: שבועית
- **חלון**: יום שני, 02:00-04:00 IST
- **תוכן**: עדכוני אבטחה (Security Patches), פאצ'ים קלים
- **השפעה**: ללא Downtime (Zero-Downtime Deploy)

### 2.2 תחזוקה רגילה (Regular)
- **תדירות**: חודשית
- **חלון**: יום שלישי הראשון של החודש, 02:00-06:00 IST
- **תוכן**: Minor versions, אופטימיזציות, שדרוגי DB קלים
- **השפעה**: עד 5 דקות Downtime

### 2.3 תחזוקה רחבה (Major)
- **תדירות**: רבעונית
- **חלון**: ראשון לא-שבת, 23:00-06:00 IST (7 שעות)
- **תוכן**: Major versions, שדרוג תשתית, מיגרציות נתונים
- **השפעה**: עד שעה Downtime

### 2.4 תחזוקת חירום (Emergency)
- **תדירות**: לפי הצורך
- **חלון**: בכל זמן (העדפה ללילה)
- **תוכן**: תיקון Critical, אירוע אבטחה
- **השפעה**: לפי המקרה

## 4. תקופות חסומות (Blackout Periods)

לא תתבצע תחזוקה (למעט חירום) ב:
- **לפני / במהלך חגי תשרי** (ראש השנה, יום כיפור, סוכות)
- **לפני / במהלך חג הפסח**
- **בשבועיים שלפני Black Friday / Cyber Monday**
- **בערב חגי לקוחות עיקריים** (כפי שהוגדרו ב-Account Plan)

## 5. הודעות

### 5.1 ערוצי הודעה
- **Status Page** — תמיד מתעדכן
- **Email** — לכל אדמין של כל tenant
- **Banner ב-UI** — 24 שעות לפני
- **Slack Webhook** — ללקוחות עם אינטגרציה

### 5.2 תבנית הודעה
```
Subject: [Scheduled Maintenance] תחזוקה רגילה — 03.03.2026

שלום,

נבצע תחזוקה מתוכננת כדי לשפר את ביצועי המערכת.

מתי: יום שלישי 03.03.2026, 02:00-06:00 IST
משך משוער: 5 דקות
השפעה: זמן Downtime קצר, ייתכן ויש להתחבר מחדש

מה משתפר:
- שדרוג גרסת DB
- אופטימיזציה של חיפוש

תודה,
צוות התשתיות
```

## 6. אישור הלקוח לחלון מותאם

לקוחות Enterprise יכולים לבקש:
- חלון תחזוקה שונה מברירת המחדל
- שמירת בלוקים על תאריכים ספציפיים (אירועים גדולים)
- Notification מיוחד למוקד שלהם

הבקשה דרך CSM.

## 7. תהליך פנימי לכל חלון

### לפני (T-7 ימים)
- [ ] תאריך אושר ופורסם ב-Status Page
- [ ] שלחנו הודעות לכל הלקוחות
- [ ] Runbook מוכן
- [ ] Rollback plan מוכן
- [ ] On-Call team משובץ
- [ ] גיבוי מלא בוצע

### בזמן התחזוקה
- [ ] Status Page מעודכן ל-"Maintenance in Progress"
- [ ] Slack War Room פתוח
- [ ] Health checks רצים ברקע
- [ ] בודקים כל שלב

### אחרי
- [ ] Smoke tests עוברים
- [ ] Status Page חזר ל-"Operational"
- [ ] Notification "תחזוקה הסתיימה"
- [ ] Monitoring למשך 2 שעות נוספות
- [ ] Post-Action Report

## 8. Rollback Policy

- כל deployment חייב Rollback Plan כתוב
- אם פעולה > 30 דקות מהמתוכנן — שיקול Rollback
- אם בעיה חמורה — Rollback מיידי, גם אם זה אומר Downtime נוסף
- Status Page מעודכן בכל החלטה כזו
